# Ground Rules

The binding rules for any scaffold session - new, vertical-slice, or brownfield-adoption. Each rule has a stable identifier (`GR-NN`) so phase gates, skills, and templates can cite the rule they enforce. The rule statements here are the canonical short form; the detailed enforcement text lives in [ai/SKILL.md](ai/SKILL.md), [START-AI.md](START-AI.md), and [support/execution-gates.md](support/execution-gates.md) as noted in the **Enforced in** column.

When two files appear to disagree, the conflict-resolution order is unchanged: [START-AI.md](START-AI.md) (session/phase routing) > [support/execution-gates.md](support/execution-gates.md) (validation commands) > [ai/SKILL.md](ai/SKILL.md) (Phase 5 routing + detail) > individual skills > templates. This file is the index; it does not override the conflict order.

## Rules

| ID | Rule | Why it matters | Enforced in |
|---|---|---|---|
| **GR-01** | Phase 1 artifacts (`.scaffold/domain-specification.yaml`, `.scaffold/UBIQUITOUS-LANGUAGE.md`, `.scaffold/DESIGN-DECISIONS.md`) are the binding source of truth. Update the artifact first, then the code. On drift, the artifact loses to code reality - update the artifact, do not silently divergence-patch. | Every later phase, every vertical slice, and every brownfield-adoption pass reasons from these files. Stale artifacts cause downstream AI sessions to invent or contradict the domain. | See [START-AI.md](START-AI.md) section Phase-1 Artifact Lifecycle Rule. Also enforced in [ai/SKILL.md](ai/SKILL.md) section Non-Negotiables and [support/HANDOFF.md](support/HANDOFF.md) section Phase-1 Artifact Currency. |
| **GR-02** | One public type per file across all generated code, including vendored `src/Packages/<Prefix>.*`. File name matches the type. Lumped files are split at generation time, not deferred. | Lumped files defeat grep/agent navigation, complicate refactoring, and hide accidental coupling. Splitting later is more expensive than splitting now. | See [ai/SKILL.md](ai/SKILL.md) section Non-Negotiables. Exception list in [skills/solution-structure.md](skills/solution-structure.md) section Non-Negotiables. Final acceptance check in [support/final-scaffold-checklist.md](support/final-scaffold-checklist.md). |
| **GR-03** | Use `.slnx` plus central package management (`Directory.Packages.props`). Do not generate legacy `.sln` or per-project `<Version=>` references. | Central package management is required for the dual-DbContext and multi-host layout the scaffold produces. The `.slnx` format is the supported solution format going forward. | See [ai/SKILL.md](ai/SKILL.md) section Non-Negotiables. Detailed package guidance in [skills/package-dependencies.md](skills/package-dependencies.md). |
| **GR-04** | Minimize third-party packages. The reference-app stack (Yarp, Scalar, FusionCache, StackExchange.Redis, Moq, NetArchTest, NBomber, Testcontainers, BenchmarkDotNet, MudBlazor, Refit, Azure.*, Aspire, plus the React allowlist when React is enabled) is the allowlist. Anything else needs developer confirmation with high-value justification, and the decision is recorded in `.scaffold/DESIGN-DECISIONS.md`. | New dependencies bring license, security, and upgrade surface. A small in-house extension promoted to a shared `src/Packages/<packagePrefix>.<Layer>` project is usually cheaper than a third party. | See [ai/SKILL.md](ai/SKILL.md) section Non-Negotiables. Detailed allowlist guidance in [skills/package-dependencies.md](skills/package-dependencies.md) section Minimize Third-Party Dependencies. |
| **GR-05** | Every external dependency declares one scaffold-time mode before Phase 5 generates code for it: `emulator`, `lazy-optional`, `no-op stub`, or `deployment-only`. The mode is recorded in `.scaffold/resource-implementation.yaml` under `externalDependencyModes`. | Without an explicit mode, the scaffold either blocks on live cloud setup or generates code that throws at runtime. The four-mode discipline keeps local boot deterministic. | See [ai/SKILL.md](ai/SKILL.md) section Key Principles. Runtime gate in [support/execution-gates.md](support/execution-gates.md) section Aspire AppHost. |
| **GR-06** | Stub external dependencies for compile/run locally. `deployment-only` still requires a no-op stub that satisfies the interface, returns safe defaults, and carries a `// TODO: [CONFIGURE]` comment. Scaffold completion never blocks on live cloud provisioning. | The scaffold proves *compilable, runnable, testable end-to-end* - not *configured for production*. Manual cloud setup is a delivery concern, not a scaffold concern. | See [ai/SKILL.md](ai/SKILL.md) section Key Principles. Final completion bar in section Scaffold Definition of Done. |
| **GR-07** | Generate code only in the user's target project directory. Never hot-edit installed `.instructions/` files mid-scaffold. Instruction gaps go to `.scaffold/INSTRUCTION-GAPS.md` at project root. | The instruction set is shared across projects. Mid-scaffold edits there cause silent drift between projects on the same harness. The gap log lets maintainers fold approved findings back into the source repo. | See [ai/SKILL.md](ai/SKILL.md) section Non-Negotiables. Maintainer fold-back in [support/UPDATE-INSTRUCTIONS.md](support/UPDATE-INSTRUCTIONS.md). |
| **GR-08** | Latest stable for SDKs, runtimes, and packages by default. Templates and instructions use `<latest-stable>` placeholders, never hard-coded versions. Pinning requires a recorded decision. | Hard-coded versions in template prose go stale on every release and silently contradict the latest-stable rule on every PR diff. | See [skills/package-dependencies.md](skills/package-dependencies.md) section Latest, Not Pinned (Global Rule). Cross-referenced from [ai/SKILL.md](ai/SKILL.md) section Non-Negotiables. |
| **GR-09** | Each phase runs in its own session. Phase 5 runs one sub-phase per session. End each session with an updated `HANDOFF.md` at project root. The next session starts from `START-AI.md` and `HANDOFF.md` only. | One-phase-per-session caps context size, makes the load-set rule meaningful, and gives the user a discrete review point between phases. | See [START-AI.md](START-AI.md) section Session Model. File-loading detail in section File Loading Rule. Resume contract in [support/HANDOFF.md](support/HANDOFF.md). |
| **GR-10** | Open questions surfaced during Phase 1 (or brownfield adoption) use the marker `[OPEN QUESTION: <single-sentence question>]` inline in the relevant `.scaffold/` artifact, mirrored to `HANDOFF.md` section Open Questions. A blocking open question halts the next phase until it is resolved or converted to a deferred decision with `Needed Before` populated. | The marker turns deferred ambiguity into a tracked artifact instead of a memory item. Phase 2 and Phase 3 gates can scan for the literal string. Nothing slips through silently. | See [ai/shared-understanding-interview.md](ai/shared-understanding-interview.md) section Clarification Quality Rules. Plan-side usage in [ai/implementation-plan.md](ai/implementation-plan.md) section Open Questions. Gate scan in [support/execution-gates.md](support/execution-gates.md) section Tooling Verification (Phase 3 Gate). Carry-over table in [support/HANDOFF.md](support/HANDOFF.md) section Open Questions. |
| **GR-11** | Scaffold Definition of Done: `dotnet build` green on the full solution. `dotnet test` green for every category the scaffold produces (with `Assert.Inconclusive` allowed for environment-gated tests). Aspire AppHost starts cleanly with all resources Running and health probes 200. Every enabled UI host starts cleanly standalone and Aspire-registered. Startup logs are quiet at Information level. Every external-dependency deferral is named in `HANDOFF.md`. Phase 1 artifacts cover every term, decision, entity, and event the scaffold introduced. | This is the single contract that determines whether the scaffold is finished. Individual phase gates feed it. Nothing else does. | See [ai/SKILL.md](ai/SKILL.md) section Scaffold Definition of Done. Acceptance form in [support/HANDOFF.md](support/HANDOFF.md) section Scaffold Acceptance. Manual checklist in [support/final-scaffold-checklist.md](support/final-scaffold-checklist.md). |
| **GR-12** | Authority map: [START-AI.md](START-AI.md) owns session boot, phase routing, and load rules. [support/execution-gates.md](support/execution-gates.md) owns validation gates and commands. [ai/SKILL.md](ai/SKILL.md) owns Phase 5 load sets and concern routing. Skills own implementation detail. Templates own generated file shape. This file (GROUND-RULES.md) is the index of the binding rules and does not change the authority order. | The authority map prevents two files from quietly claiming the same decision. New rules belong in one location and are referenced - not copied - elsewhere. | See [START-AI.md](START-AI.md) section Conflict Resolution Order. Detail in [ai/SKILL.md](ai/SKILL.md) section Non-Negotiables. |

## Adding or Changing a Rule

- Promote a rule here only when it is binding across every phase and would surprise a future maintainer if violated. Stack-specific guidance (EF Core, Aspire, Uno) belongs in the matching skill, not here.
- Each rule must have an enforcement location. If no gate, skill, or template enforces the rule today, write the enforcement before promoting the rule.
- When a rule is superseded, mark the row `superseded by GR-NN` rather than deleting it; downstream references to the old `GR-NN` stay legible.
- Renumbering is not allowed once a `GR-NN` is referenced from outside this file. Add new rules at the end of the list.
