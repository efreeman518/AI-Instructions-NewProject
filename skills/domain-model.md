# Domain Model

## Overview

Domain entities use a **rich domain model** pattern with factory creation, private setters, encapsulated validation, and `DomainResult<T>` for railway-oriented error handling. No public constructors ‚Äî all creation goes through static `Create()` methods.

## Entity Base Class

All entities inherit from `EntityBase` (from EF.Domain ‚Äî see [../support/ef-packages-reference.md](../support/ef-packages-reference.md)). Provides:
- `Guid Id` ‚Äî generated client-side, never auto-generated by DB
- `byte[] RowVersion` ‚Äî optimistic concurrency

```csharp
// From EF.Domain (verified from EF.Packages source)
public abstract class EntityBase
{
    public Guid Id { get; init; } = Guid.CreateVersion7(); // init-only, rejects Guid.Empty
    public byte[]? RowVersion { get; set; }                 // nullable, configured via .IsRowVersion() in EF config
}
```

> **CRITICAL:** `Id` is `init`-only ‚Äî it cannot be set after construction. `RowVersion` is `byte[]?` (nullable), not `byte[]`. These are verified from the [EF.Packages source](https://github.com/efreeman518/EF.Packages).

## Key Design Rules

1. **No public constructors** ‚Äî Always use `static Create()` that returns `DomainResult<T>`
2. **Private setters** ‚Äî State changes only through domain methods (`Update()`, `Add*()`, `Remove*()`)
3. **`init` for invariants** ‚Äî Properties that must not change after creation (e.g., `TenantId`) use `init`
4. **Validation on create AND update** ‚Äî `Valid()` is called in both paths
5. **DomainResult chaining** ‚Äî Use `.Bind()` and `.Map()` for composing operations:
   ```csharp
   return entity.Update(name: dto.Name)
       .Bind(e => e.AddChild(child))
       .Map(e => e);  // unwrap to parent
   ```
6. **Private parameterless constructor for EF** ‚Äî Required for materialization, never used directly
7. **Collection properties** ‚Äî Use `ICollection<T>` initialized to `[]`. Never expose `List<T>`.
8. **Convenience accessors** ‚Äî Computed properties for common lookups:
   ```csharp
   public Email? PrimaryEmail => Emails.FirstOrDefault(e => e.ContactType == ContactType.Primary)?.Email;
   ```

## Flags Enum Pattern (Domain.Shared)

Use `[Flags]` enums for entity status/features. Always start with `None = 0` and use bit shifting. See [entity-template.md](../templates/entity-template.md) for examples.

## Non-Flags Enums (Domain.Shared)

Standard enums for fixed classifications.

## Namespace Rule

> **Never infer a namespace from a folder path.** In C#, folder structure and namespace are independent. Always read the `namespace` declaration in the source file. For example, enums placed in a `Domain.Shared/Enums/` subfolder may still declare `namespace {Project}.Domain.Shared` ‚Äî not `{Project}.Domain.Shared.Enums`. Verify actual namespaces before adding `global using` statements.

## Value Objects / Shared Entities

**Owned type pattern** ‚Äî stored as columns in the parent table:
```csharp
public class DateRange  // Value object, no EntityBase
{
    public DateTimeOffset StartDate { get; init; }
    public DateTimeOffset? EndDate { get; init; }
}
```

**Polymorphic join pattern** ‚Äî links any entity to a shared entity via `{EntityType, EntityId}` discriminator. See [entity-template.md](../templates/entity-template.md).

## Join Entity Pattern (Many-to-Many)

Default: inherit from `EntityBase` with FK properties on both sides (allows adding properties later). Only use pure composite-key join when confident no additional properties will be needed. See [entity-template.md](../templates/entity-template.md) for full pattern and EF configuration.

## AuditableBase (Available but Rarely Used)

`AuditableBase<TAuditIdType>` adds `CreatedDate`, `CreatedBy`, `UpdatedDate`, `UpdatedBy` to `EntityBase`. Only use when audit fields must live ON the entity itself. If `AuditInterceptor` handles audit metadata externally (the default), inherit `EntityBase` directly.

## Tenant Entity Interface

```csharp
// From EF.Domain.Contracts
public interface ITenantEntity<TTenantId>
{
    TTenantId TenantId { get; }
}
```

Entities implementing this get automatic query filters (see multi-tenant skill).

## IRequestContext (EF.Utility)

```csharp
public interface IRequestContext
{
    Guid? TenantId { get; }
    IReadOnlyCollection<string> Roles { get; }
    string CorrelationId { get; }
    string AuditId { get; } // Used by AuditInterceptor for user identification
}
```

> **WARNING:** There is NO `.UserId` property ‚Äî use `AuditId` for user identification.

## Domain.Shared Exceptions

`NotFoundException` and `BusinessRuleException` in `Domain.Shared.Exceptions`.

## Domain Rules

Use the specification pattern for reusable business validation. Canonical placement: `src/Domain/{Project}.Domain.Model/Rules/`. Co-locate with the domain model. See [domain-rules-template.md](../templates/domain-rules-template.md) for full implementation.

## DomainResult Pattern

`DomainResult<T>` provides:
- `DomainResult<T>.Success(value)` ‚Äî successful operation
- `DomainResult<T>.Failure(errors)` ‚Äî validation failure with error list
- `.Bind(func)` ‚Äî chain operations, short-circuiting on failure
- `.Map(func)` ‚Äî transform the success value
- `.IsSuccess` / `.IsFailure` ‚Äî check state
- `.Value` ‚Äî the success value (null on failure)
- `.ErrorMessage` ‚Äî combined error string (null on success)

## Verification

After generating domain entities, confirm:

- [ ] Entity inherits from `EntityBase` (+ `ITenantEntity<Guid>` for multi-tenant ‚Äî NOT `AuditableBase`)
- [ ] Private/protected constructor + static `Create()` factory method returning `DomainResult<T>`
- [ ] All setters are `private set` ‚Äî mutations go through named methods
- [ ] Flags enum is defined with `[Flags]` attribute and `None = 0` value
- [ ] Child collections use `ICollection<T>` with controlled mutation methods (`Add/Remove`) on the entity
- [ ] `RowVersion` property exists for concurrency (configured in EF, not in entity)
- [ ] No infrastructure concerns (no EF attributes, no `DbContext`, no DTOs)
- [ ] Cross-references: Entity properties align with [data-mapping-template.md](../templates/data-mapping-template.md), EF config covers all relationships per [ef-configuration-template.md](../templates/ef-configuration-template.md)
